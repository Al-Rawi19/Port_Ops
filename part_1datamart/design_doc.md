# PortOps Data Mart — Design Document

## Fact Table Grains

**fact_container_movement**: One row per individual container move.
Chose this grain because each move is the atomic unit of work — aggregating 
to vessel call level would lose the ability to analyse cycle times, 
shift performance, and equipment utilisation at the move level.

**fact_vessel_call**: One row per vessel call (arrival/departure event).
Berth delay, stay hours, and moves variance are meaningfully computed at 
this grain.

**fact_gate_transaction**: One row per truck transaction at the gate.
Gate-in and gate-out are stored in the same row with separate date FKs,
enabling Power BI inactive-relationship analysis on gate-out date while 
keeping a simple.

## dim_date

Generated by a T-SQL WHILE loop in SQL, covering 2020-01-01 to 2026-12-31.
Range chosen: starts 2020 because CustomerHistory records go back to 
Excel serial 43831 (2020-01-01). Ends 2026-12-31 to cover 1 full year 
beyond the reporting period as a buffer.

Fiscal year attribute: April 1 start. Fiscal month 1 = April, 12 = March.

## Data Quality Issues Found

1. **Date columns stored as Excel serial floats** in source. Handled via 
   SQL conversion formula: DATEADD(DAY, CAST(col AS INT) - 2, '1900-01-01').
   The "-2" corrects for Excel's 1-based epoch and the 1900 leap year bug.

2. **effective_to column is mixed type**: numeric floats for historical end 
   dates, string '9999-12-31' for current rows. Handled with CASE WHEN in 
   the staging→mart SQL query.

3. **Trailing spaces in customer_name** (e.g., "Hapag-Lloyd  "). Handled 
   with LTRIM(RTRIM()) in the source query.

4. **is_reefer arrives as numeric** (0/1) from Excel, not a true boolean.
   Accepted as-is since SQL Server and Power BI both handle BIT = 0/1.

## Scale Trade-offs (90K → 10M+ rows)

1. Replace Full Cache on Lookup components with Partial Cache and index 
   dim_customer, dim_equipment, etc. on their natural keys.
2. Replace single Excel source with a proper staging area fed from 
   operational system APIs or a flat-file extract. Process in parallel 
   using SSIS data flow partitioning.
